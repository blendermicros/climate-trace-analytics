🌍 Your Dataset: Climate TRACE

Climate TRACE is the most comprehensive facility-level global emissions inventory ever created. It tracks greenhouse gas emissions from 2.7 million sources across 744 million assets, using satellite imagery and machine learning. The data is free, Creative Commons licensed, and updated monthly.

You'll download sector packages from climatetrace.org/data. Each package contains four CSV files that form a natural relational database:

emissions_sources.csv
Facility-level emissions. One row per source × gas × time period. The core fact table.
source_id · source_name · iso3_country · sector · subsector · gas · emissions_quantity · lat · lon · capacity · activity
country_emissions.csv
Country-level aggregates by subsector and gas. Smaller, cleaner — ideal for learning.
iso3_country · sector · subsector · gas · emissions_quantity · start_time · end_time
confidence.csv
Data quality ratings (very high → very low) per source. Teaches you to question your data.
source_id · source_type · capacity · activity · [gas]_emissions · total_co2e_100gwp
ownership.csv
Who owns each facility. Ownership chains, parent companies, share percentages.
source_id · parent_name · parent_entity_type · overall_share_percent · ownership_path

W1
SQL Fundamentals on Country-Level Emissions
10 hours · 4 sessions

Start with the smaller, cleaner country_emissions.csv. The goal this week is to think in rows and filters while exploring real emissions data. By Friday, you should be able to pull exactly the data you need from a database and have formed your first impressions of global emission patterns.

Learning Objectives

  • Load a CSV into DuckDB and write SELECT queries with WHERE, AND/OR, IN, BETWEEN, LIKE
  • Sort and limit results with ORDER BY and LIMIT
  • Understand data types and why the gas column changes the meaning of emissions_quantity
  • Handle NULL values correctly (Climate TRACE uses NULL to mean "data not yet available")
  • Explain the difference between CO2, CH4, N2O, and co2e_100yr in the dataset

📖 Core Concepts

SELECT ... FROM ... WHERE
The fundamental query. Every SQL query starts here. Practice filtering by country (iso3_country = 'DNK'), sector, and gas type.
Filtering operators
=, !=, >, IN ('DNK','SWE','NOR'), BETWEEN '2020-01-01' AND '2023-12-31', LIKE '%coal%', IS NULL
NULL semantics
In Climate TRACE: NULL = "not yet available", 0 = "this gas is not emitted." Critical distinction. NULL ≠ 0. Always use IS NULL, never = NULL.
Data types & the gas column
The same source has multiple rows — one per gas (CO2, CH4, N2O, co2e_100yr). Filtering by gas is essential or you'll double/triple count.
DuckDB CSV loading
SELECT * FROM read_csv_auto('file.csv'). Or CREATE TABLE emissions AS SELECT * FROM read_csv_auto('file.csv'). DuckDB reads CSVs natively.
Sorting & limiting
ORDER BY emissions_quantity DESC LIMIT 20. "Show me the top 20 emitting countries." The simplest useful query.
DuckDB -- Load the country emissions CSV directly in DuckDB CREATE TABLE country_emissions AS SELECT * FROM read_csv_auto('power_country_emissions.csv'); -- First look: what do we have? SELECT COUNT(*) AS total_rows, COUNT(DISTINCT iso3_country) AS countries, COUNT(DISTINCT subsector) AS subsectors, COUNT(DISTINCT gas) AS gas_types, MIN(start_time) AS earliest, MAX(end_time) AS latest FROM country_emissions; -- Denmark's power sector CO2 emissions by year SELECT iso3_country, subsector, gas, YEAR(start_time::DATE) AS year, emissions_quantity FROM country_emissions WHERE iso3_country = 'DNK' AND gas = 'co2' AND temporal_granularity = 'annual' ORDER BY year;
🧑‍🏫 Teaching note — why country_emissions first? The source-level emissions file has millions of rows and multiple rows per source per gas per time period. Starting there would overwhelm a beginner. The country_emissions file is a pre-aggregated summary — small enough to see patterns, but structured identically to the big file. It teaches the same SQL without the data engineering headaches. We introduce the full source-level data in Week 3.

W2
Aggregation, Grouping & Conditional Logic
10 hours · 4 sessions

This is where SQL goes from "selecting rows" to "answering questions." GROUP BY is the single most-used SQL pattern in analytics. This week you'll aggregate emissions across countries, sectors, and time periods to produce the kind of summary tables that show up in climate reports.

Learning Objectives

  • Use aggregate functions: COUNT, SUM, AVG, MIN, MAX — and understand how each handles NULLs
  • Group results with GROUP BY and filter groups with HAVING
  • Write CASE WHEN for categorisation (e.g., emission size buckets, sector groupings)
  • Combine multiple aggregations in a single query (conditional aggregation pattern)
  • Understand the difference between WHERE (filters rows) and HAVING (filters groups)

📖 Core Concepts

Aggregate functions
SUM(emissions_quantity) to get totals. AVG for averages. COUNT(DISTINCT iso3_country) for unique countries. All ignore NULLs except COUNT(*).
GROUP BY
Every non-aggregated column in SELECT must be in GROUP BY. Think: "for each [country], calculate [total emissions]."
HAVING vs WHERE
WHERE filters rows before grouping. HAVING filters groups after. "Countries WHERE region = 'Europe'" vs "Countries HAVING SUM(emissions) > 1000000".
CASE WHEN
SQL's if/else. Create emission size categories, flag countries above thresholds, or do conditional aggregation (the secret weapon).
SQL — Conditional aggregation on emissions -- "For each country, how much comes from electricity vs heat generation?" SELECT iso3_country, SUM(CASE WHEN subsector = 'electricity-generation' THEN emissions_quantity ELSE 0 END) AS electricity_co2e, SUM(CASE WHEN subsector = 'heat-plants' THEN emissions_quantity ELSE 0 END) AS heat_co2e, SUM(emissions_quantity) AS total_co2e FROM country_emissions WHERE gas = 'co2e_100yr' AND temporal_granularity = 'annual' AND YEAR(start_time::DATE) = 2023 GROUP BY iso3_country HAVING SUM(emissions_quantity) > 1000000 ORDER BY total_co2e DESC LIMIT 20;

JOINs & Multi-Table Emissions Analysis
10 hours · 4 sessions

Now we bring in the full source-level data. This is where it gets real: you'll connect emissions data with ownership and confidence data using JOINs. The questions get much more interesting — "Which companies own the dirtiest power plants in Europe?" requires data from multiple tables.

🧑‍🏫 Teaching note — the jump in complexity The source-level emissions file is much larger than country_emissions. Each facility has multiple rows (one per gas per time period). The ownership file has its own structure, with multiple owners per source. This week is intentionally harder — JOINs are where SQL becomes a data integration language, and Climate TRACE's multi-file structure is the perfect training ground.

Learning Objectives

  • Understand and use INNER JOIN, LEFT JOIN, and their behaviour with one-to-many relationships
  • Join emissions_sources to ownership and confidence tables on source_id
  • Write subqueries in WHERE, FROM, and SELECT clauses
  • Debug accidental row multiplication when joining multi-row tables
  • Understand the source_id as the key that links the Climate TRACE universe

📖 Core Concepts

INNER JOIN
Only sources that have ownership data. Use: "power plants WHERE we know who owns them."
LEFT JOIN
All sources + ownership where available, NULLs where not. The analyst's workhorse. "Show all plants, flag those with unknown owners."
Row multiplication trap
One source can have multiple owners (shared ownership). JOIN without care = inflated emissions. Always check COUNT before and after.
Subqueries
A query inside another. "Sources whose emissions are above the average" needs a subquery for the average. Think of them as intermediate calculations.
SQL — Joining emissions with ownership -- "Top 20 parent companies by total CO2 emissions from European power plants" SELECT o.parent_name, COUNT(DISTINCT e.source_id) AS num_facilities, ROUND(SUM(e.emissions_quantity) / 1e6, 2) AS total_co2_mt FROM emissions_sources e INNER JOIN ownership o ON e.source_id = o.source_id WHERE e.gas = 'co2' AND e.temporal_granularity = 'annual' AND YEAR(e.start_time::DATE) = 2023 AND e.iso3_country IN ('DNK','DEU','POL','SWE','NOR','FIN') GROUP BY o.parent_name ORDER BY total_co2_mt DESC LIMIT 20;

W4
Advanced SQL: CTEs & Window Functions
10 hours · 4 sessions

This is the week that separates "I know SQL" from "I'm good at SQL." CTEs make complex queries readable; window functions let you calculate rankings, trends, and running totals without leaving SQL. Every senior analyst job listing tests for these skills.

Learning Objectives

  • Write Common Table Expressions (CTEs) with WITH to break queries into named, readable steps
  • Use window functions: ROW_NUMBER, RANK, DENSE_RANK, NTILE for ranking
  • Use LAG and LEAD for year-over-year emissions comparisons
  • Calculate running totals and cumulative percentages with SUM() OVER (ORDER BY ...)
  • Understand PARTITION BY as "GROUP BY without collapsing rows"

📖 Core Concepts

CTEs (WITH clause)
Named temporary results. Each CTE is a "paragraph" in your SQL essay. Chain them: raw data → filtered → aggregated → ranked.
Window functions
FUNCTION() OVER (PARTITION BY ... ORDER BY ...). Calculates across a "window" of rows without collapsing them. The analyst's power tool.
LAG / LEAD
Access previous/next row. LAG(emissions, 1) = "last year's emissions." Foundation of trend analysis.
Ranking functions
ROW_NUMBER (no ties), RANK (ties skip), NTILE(4) = quartiles. "Rank power plants within each country by emissions."
SQL — CTE + Window Function: Emissions trend analysis WITH annual_emissions AS ( SELECT iso3_country, YEAR(start_time::DATE) AS year, SUM(emissions_quantity) AS total_co2e FROM country_emissions WHERE gas = 'co2e_100yr' AND temporal_granularity = 'annual' AND iso3_country IN ('DNK','SWE','NOR','FIN') GROUP BY iso3_country, YEAR(start_time::DATE) ), with_trend AS ( SELECT *, LAG(total_co2e) OVER ( PARTITION BY iso3_country ORDER BY year ) AS prev_year, RANK() OVER ( PARTITION BY year ORDER BY total_co2e DESC ) AS rank_in_year FROM annual_emissions ) SELECT iso3_country, year, ROUND(total_co2e / 1e6, 2) AS co2e_million_tonnes, ROUND((total_co2e - prev_year) / prev_year * 100, 1) AS yoy_change_pct, rank_in_year FROM with_trend ORDER BY year, rank_in_year;

W5
Python Data Wrangling & Building the Database
10 hours · 4 sessions

SQL handles structured data. But the raw Climate TRACE CSVs need cleaning first — inconsistent formats, wide vs long reshaping, multiple files that need combining, encoding edge cases. This week you learn pandas as the bridge between raw downloads and a queryable database.

Learning Objectives

  • Read Climate TRACE CSVs into pandas DataFrames and audit them (.shape, .dtypes, .describe(), .isnull())
  • Clean: parse dates, fix types, handle NULLs with strategy, standardise column names
  • Merge multiple sector CSVs into a unified emissions table
  • Reshape data with melt (wide → long) and pivot_table (long → wide)
  • Build a complete DuckDB database from cleaned DataFrames — the database you'll use for the rest of the curriculum

📖 Core Concepts

The 5-Command Audit
.shape, .dtypes, .describe(), .isnull().sum(), .head(20). Run these on every new CSV before doing anything else.
Cleaning patterns
.str.strip(), pd.to_datetime(), pd.to_numeric(errors='coerce'), .fillna(), .drop_duplicates(). Build a reusable clean() function.
Combining files
pd.concat() to stack multiple sector CSVs. pd.merge() for SQL-like JOINs. Use validate='one_to_many' to catch bugs.
Reshaping
pd.melt() turns wide tables into long (good for SQL/plotting). .pivot_table() does the reverse. Climate TRACE is already long format — you'll practice both directions.
SQL ↔ Python
duckdb.connect() → conn.execute("CREATE TABLE x AS SELECT * FROM df"). Push cleaned DataFrames into DuckDB. Query them with SQL. Best of both worlds.
Documentation
Use Jupyter Markdown cells to document every cleaning decision. "I dropped 342 rows where emissions_quantity was NULL because..." Future-you will thank present-you.
